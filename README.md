# 🧪 Symbiosis Research Sandbox & PoC Playground
### *Эра Пост-Квантовой Криптографии & Теории Игр в Ликвидном Стейкинге (Интерактивная Симуляция)*

> ⚠️ **ВАЖНОЕ ПРЕДУПРЕЖДЕНИЕ (DISCLAIMER):**  
> Этот проект является **экспериментальным исследовательским стендом (Proof-of-Concept Sandbox)** и образовательным симулятором. Он **НЕ** предназначен для развертывания в промышленной сети (Production Mainnet), коммерческой эксплуатации или привлечения реальных инвестиционных средств. Все задействованные механизмы пост-квантовых подписей, ZK-доказательств и теории игр представлены в виде **интерактивных математических моделей и клиент-серверных эмуляций**.

## 🔍 Назначение и Архитектурный Контекст

**Symbiosis Sandbox** был разработан как визуально-интерактивный симулятор для исследования гипотетических блокчейн-механик будущего. Он позволяет наглядно изучить, как ликвидный стейкинг и протоколы валидации могут адаптироваться к вызовам пост-квантовой эпохи и оптимизировать экономические штрафы (слэшинг) с помощью теории игр.

Проект состоит из двух уровней:
1. **Теоретическая смарт-контрактная база (Solidity)**: Набор оптимизированных контрактов взаимодействия токена стейкинга ERC-20 (`SYM`), ликвидного токена (`sSYM`), реестра валидаторов и простейшего реестра ZK-доказательств.
2. **Интерактивный симулятор пользовательского интерфейса (React/Vite & Node.js Daemon)**: Позволяет разработчикам и исследователям тестировать сценарии поведения валидаторов, реагировать на двойное подписание блоков L2 в режиме реального времени и генерировать имитационные криптографические доказательства.

## ⚙️ Реальность против Спецификаций (Honest Technical Audit)

Для исключения заблуждений ниже приводится объективный и честный разбор заявленных технологических аспектов:

### 1. Пост-квантовые подписи Falcon-512 и "Precompile 0xF9"
* **Заявленная теория**: Использование решетчатого алгоритма криптографии Falcon-512 (стандарт NIST) на уровне виртуальной машины (EVM) вместо уязвимого для квантовых алгоритмов ECDSA (secp256k1).
* **Техническая реальность**: В публичных блокчейн-сетях (Ethereum Mainnet, Base Sepolia, Arbitrum Sepolia и др.) **не существует precompile-контракта `0xF9`**. Проект решетчатых подписей в EVM (EIP-7619) на момент написания находится только в статусе чернового обсуждения.
* **Как это решено в песочнице**: В смарт-контрактах проекта заложены заглушки, эмулирующие валидность подписей в тестовых целях. Настоящая проверка подписи в продакшен-сети потребовала бы кастомной модификации клиента ноды (например, форка Geth/Reth) или написания сверхдорогих по газу WASM/Solidity-верификаторов. Здесь это представлено в виде **эмулятора**.

### 2. "Nash Consensus" (Равновесие Нэша)
* **Заявленная теория**: Узлы конкурируют на основе математической теории игр, регулирующей стейкинг и защищающей сегменты от картельных сговоров с помощью ловушек (Red Herrings).
* **Техническая реальность**: Консенсус сети по-прежнему полностью обеспечивается базовым слоем (L1 Ethereum или L2 Rollup), в котором развернут контракт. "Консенсус Нэша" в данном случае — это **прикладной токеномический смарт-контракт (`NashConsensusRegistry.sol`)**, регулирующий правила внесения залога, распределения наград и удаления вредоносных валидаторов. Это экономическая надстройка, а не сетевой протокол консенсуса (такой как PBFT, Tendermint или Gasper).

### 3. ZK-Cops и моментальный слэшинг
* **Заявленная теория**: Мгновенный децентрализованный аудит действий валидаторов с помощью ZK-доказательств отсутствия саботажа.
* **Техническая реальность**: Настоящий ZK-Snark прувер требует наличия скомпилированных математических схем вычислений (Circom/Leo), генерации доверенной настройки (trusted setup, файлов `.ptau`) и тяжеловесных оффчейн-вычислений.
* **Как это решено в песочнице**: Верификация в dApp воссоздает пошаговую симуляцию компиляции R1CS-ограничений (Multiplier/Groth16) и отправляет демо-транзакцию подтверждения репутации в MetaMask. Это наглядная интерактивная демонстрация логики, спроектированная для демонстрации потенциального UX-взаимодействия будущих ZK-клиентов.

## 🎨 Возможности Интерактивного Стенда (Sandbox Interface)

Приложение предлагает полноценную интерактивную среду для симуляции сети:
* **Валидаторы и Монитор активности**: Визуализация распределенных нод, их текущий стейкинг, репутация и генерация блоков L2.
* **Защита в Смарт-контрактах (Live Audit)**: Интерактивный аудит безопасности, демонстрирующий, как контракты выдерживают атаки повторного входа (Reentrancy), проверяют блокировки вывода средств и страхуют депозиты.
* **Simulated ZK-Cops Prover**: Вкладка, демонстрирующая шаги компиляции пруверов для Groth16.
* **Мобильный PWA Light-Client**: Интегрированная демонстрация легкого узла, сканирующего сеть в поисках византийских ошибок (например, двойной подписи блока). С возможностью включения автоматической отправки транзакций слэшинга ("whistleblowing") через MetaMask.
* **DAO governance**: Симуляция подачи голосов и распределения весов голосования.

## 🚀 Инструкции по Локальному Запуску и Тестированию

Если вы изучаете код проекта или хотите запустить симуляцию локально:

### 1. Установка и зависимости
```bash
npm install
```

### 2. Локальная компиляция контрактов
Скомпилируйте контракты с помощью встроенного в проект Hardhat (без реального деплоя в сеть):
```bash
npx hardhat compile --force
```

### 3. Запуск тестов
В проекте написана обширный набор юнит-тестов, проверяющий математические граничные условия стейкинга:
```bash
npx hardhat test
```

### 4. Локальный запуск симулятора
Вы можете запустить dApp-панель локально на порту 3000:
```bash
npm run dev
```

## 🤝 Заключение & Перспективы Исследования

Проект **Symbiosis** выполняет важнейшую образовательную и исследовательскую функцию. Он объединяет в одном интерфейсе несколько передовых концептов: пост-квантовую подпись (Falcon-512), слэшинг на основе теории игр и ZK-доказательства. 

В качестве прототипа он наглядно показывает, с какими интерфейсными и архитектурными сложностями столкнутся разработчики реального Web3-пространства при приближении квантовой угрозы и как P2P-клиенты могут взаимодействовать с детекторами аномалий. Любое дальнейшее развитие проекта должно быть сфокусировано на интеграции реальных оптимизированных WASM-верификаторов Falcon и полноценных схем Circom для оффчейн-генерации подлинных Groth16 доказательств.
